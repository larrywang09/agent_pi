# 第 1 章 项目总览与架构设计

## 1.1 项目背景：pi 是什么？

**pi** 是一个开源的、可扩展的 AI 编程代理（Coding Agent）命令行工具。它由 [Mario Zechner](https://github.com/badlogicgames)（又名 badlogicgames）创建，项目全称为 **pi agent harness**，代码托管在 GitHub 的 `earendil-works/pi-mono` 组织下。

与市面上其他 AI Coding Agent 相比，pi 有以下独特的定位：

- **终端原生（Terminal-Native）**：基于 Node.js 的 TUI（Terminal UI）构建，而不是 VSCode 插件或 Web 界面。这意味着你可以在任何终端中使用它——本地、SSH、tmux、甚至手机上（Termux）。
- **可扩展（Extensible）**：提供完整的扩展系统（Extension System），支持自定义工具、命令、UI 组件、生命周期钩子、自定义 LLM Provider 等。
- **多 Provider 支持**：内置对 OpenAI（Completions/Responses）、Anthropic（Messages）、Google（Gemini）、AWS Bedrock、Mistral、GitHub Copilot 等主流 LLM API 的统一接入。
- **会话管理**：支持会话持久化（JSONL 格式）、分支（Fork）、上下文压缩（Compaction），可以随时切换和恢复对话历史。
- **纯 TypeScript**：完全用 TypeScript 编写，基于 Node.js 22+，使用 ESM 模块系统。

pi 的官方网站是 [pi.dev](https://pi.dev)，提供在线文档和演示。

## 1.2 Monorepo 架构：4 个包的分工与依赖

pi 采用 **monorepo** 结构，由 4 个独立的 npm 包组成，按照从底层到顶层的顺序排列：

```
┌─────────────────────────────────────────────┐
│  @earendil-works/pi-coding-agent   (v0.75.3)│  ← 应用层：CLI、SDK、扩展
├─────────────────────────────────────────────┤
│  @earendil-works/pi-agent-core     (v0.75.3)│  ← 代理层：Agent 运行时
├─────────────────────────────────────────────┤
│  @earendil-works/pi-ai             (v0.75.3)│  ← AI 层：统一 LLM API
├─────────────────────────────────────────────┤
│  @earendil-works/pi-tui            (v0.75.3)│  ← 终端层：TUI 框架
└─────────────────────────────────────────────┘
```

所有包采用 **lockstep 版本管理**，即每次发布时所有包使用相同的版本号。

@earendil-works/pi-tui     ← 无内部依赖（only get-east-asian-width, marked）

## 1.3 从终端到 LLM 的数据流全景

理解 pi 的最佳方式是跟随一次典型的交互流程：用户在终端中输入提示，pi 调用 LLM 并返回结果。下面是一次完整的请求-响应数据流：

```
┌─────────────────────────────────────────────────────────────────────┐
│                       用户终端 (Terminal)                            │
└──────────────────────────────┬──────────────────────────────────────┘
                               │ 键盘输入 (ANSI 转义序列)
                               ▼
┌─────────────────────────────────────────────────────────────────────┐
│  @earendil-works/pi-tui                                            │
│                                                                     │
│  1. ProcessTerminal 读取 stdin → 原始字节数据                        │
│  2. StdinBuffer 批量分割 + Kitty 协议解析                            │
│  3. keys.ts: matchesKey() 判断按键类型                               │
│  4. Editor 组件接收输入 → 渲染文本行                                  │
│  5. 用户按 Enter → TUI 调用 InputListener → 触发提交事件              │
│  6. TUI.requestRender() → 差分渲染引擎计算差异 → 同步输出到终端       │
└──────────────────────────────┬──────────────────────────────────────┘
                               │ 提交的提示文本
                               ▼
┌─────────────────────────────────────────────────────────────────────┐
│  @earendil-works/pi-coding-agent (InteractiveMode)                  │
│                                                                     │
│  1. InteractiveMode.onSubmit() → 构造 AgentMessage                  │
│  2. AgentSession.enqueueUserMessage() → 排队等待处理                 │
│  3. AgentSession.runPrompt() → 触发 Agent.steer()/prompt()          │
│  4. Agent 状态切换为 isStreaming = true                              │
└──────────────────────────────┬──────────────────────────────────────┘
                               │ AgentMessage
                               ▼
┌─────────────────────────────────────────────────────────────────────┐
│  @earendil-works/pi-agent-core (Agent Loop)                         │
│                                                                     │
│  1. runAgentLoop() 启动循环                                          │
│  2. emit({ type: "turn_start" })                                    │
│  3. streamAssistantResponse():                                       │
│     a. transformContext() → 上下文窗口裁剪                            │
│     b. convertToLlm() → AgentMessage[] → Message[]                  │
│     c. streamSimple() → 调用 @earendil-works/pi-ai                  │
│  4. 流式收到 AssistantMessage 事件 → emit message_update 事件        │
│  5. 检测到 toolCall → executeToolCalls()                             │
│     a. prepareToolCall → beforeToolCall Hook                         │
│     b. executePreparedToolCall → 并行/顺序执行                        │
│     c. finalizeExecutedToolCall → afterToolCall Hook                 │
│  6. 循环回到 step 3，直到 stopReason = "stop"                        │
│  7. 检查 steering/followUp 队列 → 继续或结束                         │
└──────────────────────────────┬──────────────────────────────────────┘
                               │ Message[] + Model
                               ▼
┌─────────────────────────────────────────────────────────────────────┐
│  @earendil-works/pi-ai                                              │
│                                                                     │
│  1. stream(model, context, options):                                 │
│     a. getApiProvider(model.api) → 从 Registry 查找 Provider        │
│     b. provider.stream(model, context) → 返回 EventStream           │
│  2. 懒加载 Provider 模块：loadAnthropicProviderModule() 等           │
│  3. Provider 将 Message[] 转换为 Provider 特定的 API 请求格式        │
│  4. 发送 HTTP 请求到 LLM API (SSE / WebSocket)                      │
│  5. 解析响应流：text_delta / thinking_delta / toolcall_delta         │
│  6. 收集 usage 信息：input tokens, output tokens, cache 命中         │
│  7. 组装最终 AssistantMessage → 通过 EventStream 推送给 Agent       │
└──────────────────────────────┬──────────────────────────────────────┘
                               │ HTTP SSE 流
                               ▼
                    ┌─────────────────────┐
                    │  LLM Provider API   │
                    │ (OpenAI/Anthropic/  │
                    │  Google/Bedrock...) │
                    └─────────────────────┘
```

### 关键数据流总结

从终端输入到 LLM 返回，数据经历了以下转换：

```
用户按键 (ANSI bytes)
  → keys.ts: 解析为 KeyEvent (如 ctrl+enter)
  → Editor: 文本编辑
  → InteractiveMode: onSubmit → AgentMessage
  → AgentSession: queue & runPrompt
  → Agent: runAgentLoop → convertToLlm → Message[]
  → pi-ai: stream() → provider.stream() → HTTP SSE
  → LLM API: 推理
  → HTTP SSE 流返回
  → pi-ai: 解析为 AssistantMessageEvent (text_delta, toolcall_delta...)
  → Agent Loop: 收到事件 → emit message_update
  → InteractiveMode: 监听事件 → Markdown 组件渲染
  → TUI: 差分渲染 → 写入 stdout → 用户看到流式输出
```

### TypeScript 约束

项目的 `AGENTS.md` 中明确规定了只能使用 **erasable TypeScript 语法**——即编译后不会产生运行时 JavaScript 代码的语法。具体限制：

- ❌ 禁止使用：`enum`、`namespace`/`module`、`import =`、`export =`、构造器参数属性
- ✅ 允许：`interface`、`type`、泛型、`as` 类型断言、`satisfies` 关键字等

这一约束意味着项目使用 TypeScript **仅作为类型检查器**，实际的运行时行为完全由原始 JavaScript（strip-only）决定。

## 1.5 设计哲学：分层解耦

pi 的 4 层架构体现了几个核心设计原则：

### 1.5.1 关注点分离（Separation of Concerns）

每一层只关注一个核心职责：

- **pi-tui**：只关心终端渲染和用户交互。不知道 LLM、Agent 或编程工具的存在。
- **pi-ai**：只关心 LLM API 的抽象和统一。不知道 Agent 循环或 UI 的存在。
- **pi-agent-core**：只关心 Agent 运行时逻辑。通过 `streamFn` 抽象调用 LLM，通过事件接口输出状态。
- **pi-coding-agent**：组合上述所有包，添加 CLI 入口、扩展系统、会话管理。

### 1.5.2 面向接口而非实现

层与层之间通过清晰的接口通信：

```typescript
// pi-tui 的核心接口
interface Component {
  render(width: number): string[];
  handleInput?(data: string): void;
  invalidate?(): void;
}

// pi-ai 的核心抽象
type StreamFunction<TApi, TOptions> = (
  model: Model<TApi>,
  context: Context,
  options?: TOptions,
) => AssistantMessageEventStream;

// pi-agent-core 的核心抽象
interface AgentTool<TParameters extends TSchema, TDetails> extends Tool<TParameters> {
  execute: (
    toolCallId: string,
    params: Static<TParameters>,
    signal?: AbortSignal,
    onUpdate?: AgentToolUpdateCallback<TDetails>,
  ) => Promise<AgentToolResult<TDetails>>;
}
```

### 1.5.3 可替换性（Replaceability）

每一层都可以独立替换：

- **pi-tui** 的 `Terminal` 接口可以被 `VirtualTerminal` 替换用于测试，也可以被其他终端实现替换。
- **pi-ai** 的 Provider 系统通过 `api-registry.ts` 的可插拔注册机制，支持动态添加新的 LLM Provider。
- **pi-agent-core** 的 `Agent` 类接收可自定义的 `streamFn`、`convertToLlm`、`beforeToolCall`、`afterToolCall` 等回调。
- **pi-coding-agent** 的扩展系统允许用户在运行时注入自定义行为和 UI。

### 1.5.4 事件驱动（Event-Driven）

整个系统围绕事件驱动架构构建：

```
pi-ai: AssistantMessageEvent 流
  → pi-agent-core: AgentEvent 事件
    → pi-coding-agent: Extension 生命周期事件
      → pi-tui: 渲染更新
```

每一层的事件是上一层的输入，这种管道式的设计使得系统高度解耦且易于调试。

## 1.7 本章小结

本章我们了解了 pi 项目的全貌：

1. **项目定位**：一个终端原生的、可扩展的 AI 编程代理 CLI 工具。
2. **4 层架构**：从底层的 TUI 框架到顶层的 Coding Agent 应用，每层各司其职。
3. **数据流**：用户的终端输入经过 TUI → InteractiveMode → Agent Loop → LLM Provider → LLM API，最终以事件流的形式返回并渲染到屏幕。
4. **技术栈**：Node.js 22+、纯 TypeScript ESM、erasable syntax、typebox schema。
5. **设计哲学**：关注点分离、面向接口、可替换性、事件驱动。

从下一章开始，我们将从最底层的 **@earendil-works/pi-tui** 开始，逐层深入每个包的源码细节。
