# 第 9 章 回顾与展望

## 9.1 架构回顾

在本书中，我们从底层到顶层，完整地剖析了 pi agent harness 项目的 4 层架构。现在让我们回顾每层的核心贡献和设计得失。

### 分层架构总览

```
┌─────────────────────────────────────────────────┐
│  @earendil-works/pi-coding-agent                │
│  ├── 应用层：CLI、扩展系统、会话管理           │
│  ├── 依赖：pi-tui + pi-ai + pi-agent-core      │
│  └── ≈300 个源文件                              │
├─────────────────────────────────────────────────┤
│  @earendil-works/pi-agent-core                  │
│  ├── 代理层：状态机、循环、工具引擎            │
│  ├── 依赖：pi-ai                                │
│  └── ≈7 个核心源文件 + harness 辅助模块          │
├─────────────────────────────────────────────────┤
│  @earendil-works/pi-ai                          │
│  ├── AI 层：统一 LLM API、Provider 注册         │
│  ├── 依赖：外部 SDK 无内部依赖                  │
│  └── ≈10 个核心源文件 + 9 个 Provider            │
├─────────────────────────────────────────────────┤
│  @earendil-works/pi-tui                         │
│  ├── 终端层：差分渲染、组件体系                │
│  ├── 依赖：仅 marked + get-east-asian-width      │
│  └── ≈14 个源文件                               │
└─────────────────────────────────────────────────┘
```

### 设计的得与失

**成功之处：**

1. **极致的关注点分离**：每一层只做一件事，且对其他层的存在一无所知。
   - pi-tui 不知道 LLM 的存在
   - pi-ai 不知道 UI 的存在
   - pi-agent-core 通过 `streamFn` 抽象化 LLM 调用
   - 两层之间的耦合度极低，可以独立演进

2. **类型安全性**：通过 `Api` 类型参数（`Model<"openai-completions">`），在编译时就能确保 Provider 和 API 格式匹配。

3. **可扩展性**：扩展系统 + API Registry 的双重可插拔机制，使得 pi 可以无缝集成新的 Provider、工具和 UI 组件。

4. **事件驱动架构**：从 LLM 的 `AssistantMessageEvent`，到 Agent 的 `AgentEvent`，到扩展系统的生命周期事件——统一的异步事件流贯穿整个系统。

5. **增量渲染性能**：TUI 的差分渲染策略确保即使是高频的流式输出，也不会产生闪烁或性能问题。

**可以改进的地方：**

1. **测试覆盖率**：虽然用了 Faux Provider，但部分包的测试覆盖率仍然不高，特别是 TUI 组件的测试较少。

2. **Agent 循环的复杂性**：`agent-loop.ts` 的双层循环逻辑（内层处理工具调用，外层处理 FollowUp）虽然灵活，但理解起来有一定成本。

3. **消息转换的复杂度**：`transformMessages.ts` 中处理各种 Provider 的消息格式差异，随着 Provider 增多，复杂度呈线性增长。

4. **启动时间**：虽然 Provider 是懒加载的，但扩展系统的扫描和加载在扩展数量多时仍可能影响启动速度。

## 9.3 未来方向

### 9.3.1 多 Agent 协作

当前 pi 是一个**单 Agent** 系统。未来可以考虑支持多 Agent 协作架构：

```
用户问题
  ↓
协调 Agent (Orchestrator)
  ├── 代码分析 Agent → 阅读文件、搜索代码
  ├── 编写代码 Agent → 实现功能
  ├── 测试 Agent → 编写测试、运行测试
  └── 审查 Agent → Code Review、安全检查
```

每个子 Agent 可以有不同的模型配置（如大型模型用于规划，小型模型用于简单工具调用）。

### 9.3.2 IDE 集成

尽管 pi 是终端原生的，但通过其 RPC 模式，可以构建 VSCode、JetBrains、Zed 等 IDE 的扩展：

```
VSCode Extension (TypeScript)
  │
  └── JSON-RPC over stdio
        │
        pi --mode rpc
        ├── 用户消息 → prompt
        ├── 文件内容 → read tool
        ├── 代码修改 → edit/write tool
        └── 流式输出 → 事件回调
```

`examples/rpc-extension-ui.ts` 已经展示了一个 VSCode 风格的 UI 原型。

### 9.3.3 更多 Provider 支持

pi 的 Provider 系统已经支持 30+ 个 Provider，但仍有扩展空间：

- **国内 Provider**：百度文心、阿里通义、字节豆包等的接入
- **本地模型**：通过 Ollama/LM Studio 集成本地运行的模型
- **企业自定义**：通过扩展系统的 `registerProvider` API 注册企业内部的自定义 LLM

### 9.3.4 AI Function Calling 的演进

随着 LLM 能力的增强，工具调用（Function Calling / Tool Use）的范式也在演进：

- **流式工具调用**：已在 pi 中得到支持（通过 `partial-json` 实时解析）
- **嵌套工具调用**：一个工具的结果可以触发另一个工具
- **Agent 间工具调用**：Agent 之间共享工具结果
- **持续工具调用**：工具可以保持连接（如 WebSocket），持续推送结果

### 9.3.5 技能生态

pi 的技能系统（Skills）已经支持从 `SKILL.md` 文件加载可复用的 AI 技能提示词。未来可以：

- 建立技能市场（Skill Marketplace）
- 支持技能版本管理
- 技能之间的组合和链接
- AI 推荐的上下文技能

## 9.5 最后的思考

pi 是一个展示了 **如何用 TypeScript 构建生产级 AI Agent 系统** 的杰出项目。它的设计理念——分层解耦、事件驱动、可插拔扩展、终端原生——不仅仅适用于 AI 代理，也是任何复杂 CLI 应用的设计范式。

通过阅读本书和 pi 的源码，你收获的不仅仅是"如何用 pi"，更是：

1. **终端编程的艺术**：差分渲染、ANSI 序列、键盘协议
2. **LLM 抽象的技巧**：统一的事件协议、可插拔 Provider
3. **Agent 运行时的设计**：状态机、消息队列、Hook 链
4. **可扩展系统架构**：Extension API、20+ 生命周期钩子
5. **工程实践**：Monorepo、Biome、Vitest、Faux Provider 测试

希望你能将所学应用到自己的项目中，无论是一个新的 AI 代理、一个终端工具、还是一个可扩展的 CLI 应用。
