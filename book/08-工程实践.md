# 第 8 章 工程实践

## 8.1 开发环境与工具链

### 8.1.1 Node.js 版本要求

pi 要求 **Node.js >= 22.19.0**，这是一个较新的版本要求，主要利用了 Node 22 的 ESM 稳定性和性能改进。

### 8.1.2 TypeScript 配置详解

pi 的 TypeScript 配置非常独特，它使用 **erasableSyntaxOnly** 模式（`tsgo` 编译）：

```json
// tsconfig.base.json
{
    "compilerOptions": {
        "target": "ES2022",
        "module": "Node16",
        "lib": ["ES2022"],
        "strict": true,
        "erasableSyntaxOnly": true,    // ← 关键：只允许可擦除语法
        "moduleResolution": "Node16",
        "allowImportingTsExtensions": true,  // ← 允许 .ts 扩展名导入
        "rewriteRelativeImportExtensions": true,  // ← 编译时重写为 .js
        "declaration": true,
        "declarationMap": true,
        "sourceMap": true,
    }
}
```

**erasableSyntaxOnly** 的作用：
- 禁止 `enum`（编译后产生 JavaScript 代码）
- 禁止 `namespace`/`module`（编译后产生代码）
- 禁止构造器参数属性（如 `constructor(public name: string)`）
- 禁止 `import =`、`export =`
- 允许 `interface`、`type`、泛型、类型断言、`satisfies` 等仅类型语法

```typescript
// ✅ 允许的写法
interface User { name: string; }
type ID = string | number;
const x = value satisfies SomeType;

// ❌ 禁止的写法
enum Color { Red, Green, Blue }  // 会产生 JS 代码
class Foo { constructor(public name: string) {} }  // 参数属性
```

### 8.1.3 Biome 配置

pi 使用 **Biome** 替代传统的 ESLint + Prettier 组合。配置位于 `biome.json`：

```json
{
    "linter": {
        "rules": {
            "recommended": true,
            "style": {
                "useConst": "error",           // 优先使用 const
                "useNodejsImportProtocol": "off"  // 不强制 node: 前缀
            },
            "suspicious": {
                "noExplicitAny": "off",        // 允许 any（但应尽量少用）
            }
        }
    },
    "formatter": {
        "indentStyle": "tab",
        "indentWidth": 3,
        "lineWidth": 120
    },
    "files": {
        "includes": [
            "packages/*/src/**/*.ts",
            "packages/*/test/**/*.ts",
            "packages/coding-agent/examples/**/*.ts"
        ]
    }
}
```

Monorepo 通过 npm workspaces 管理：

// 根 package.json

## 8.2 构建流程

pi 的构建使用 `tsgo`（零配置 TypeScript 编译），替代传统的 `tsc`：

```bash
# 每个包的 build 脚本
tsgo -p tsconfig.build.json
```

`tsgo` 的特点：
- 自动处理 `.ts` → `.js` 扩展名重写
- 不执行类型检查（类型检查由 `npm run check` 中的 `tsgo --noEmit` 负责）
- 更快的构建速度

完整的构建顺序（由根 `package.json` 的 `build` 脚本控制）：

```
1. cd packages/tui    && npm run build   ← 先构建最底层
2. cd packages/ai     && npm run build   ← 构建前还生成模型数据
3. cd packages/agent  && npm run build
4. cd packages/coding-agent && npm run build  ← 最后构建顶层
   └── 构建后复制资源（主题、图片、HTML 模板等）
```

pi-ai 的构建多了一个步骤——**模型数据生成**：

```bash
npm run generate-models       # 从 provider 源抓取模型列表
npm run generate-image-models # 生成图像模型数据
tsgo -p tsconfig.build.json   # 编译
```

生成的模型数据写入 `models.generated.ts`，这个文件不应该被手动编辑。

## 8.3 测试体系

### 8.3.1 Vitest 配置

pi 使用 Vitest 作为测试框架，每个包有独立的配置：

```typescript
// packages/agent/vitest.config.ts
import { defineConfig } from "vitest/config";

export default defineConfig({
    test: {
        globals: true,
        environment: "node",
        testTimeout: 30000,  // LLM 调用可能较慢
    },
});
```

运行测试：

```bash
# 运行所有测试（从根目录）
./test.sh

# 运行特定测试文件（从包目录）
npx vitest --run test/agent-loop.test.ts

# 使用 Faux Provider 的测试
npx vitest --run test/faux-provider.test.ts
```

### 8.3.2 Faux Provider 测试模式

pi-ai 提供了一个 **Faux Provider**（假 Provider），允许在**不调用真实 LLM API** 的情况下测试完整的工作流。这是 pi 测试体系的核心。

```typescript
import { registerFauxProvider, fauxAssistantMessage, complete, Type } from "@earendil-works/pi-ai";

// 注册 Faux Provider
const registration = registerFauxProvider();

// 设置模拟响应
registration.setResponses([
    fauxAssistantMessage("Hello, this is a mock response!"),
    // 支持 tool calls
    fauxAssistantMessage([
        fauxText("I'll search for that."),
        fauxToolCall("search", { query: "typescript patterns" }),
    ]),
    // 支持 thinking
    fauxAssistantMessage([
        fauxThinking("Let me think about this..."),
        fauxText("I think the answer is 42."),
    ]),
]);

// 获取模拟模型
const model = registration.getModel();

// 正常调用 complete/stream，但不会真正调用 LLM
const response = await complete(model, {
    systemPrompt: "Be concise.",
    messages: [{ role: "user", content: "Hello!", timestamp: Date.now() }],
});

// 验证
expect(response.content[0]).toEqual({ type: "text", text: "Hello, this is a mock response!" });
expect(registration.state.callCount).toBe(1);
```

Faux Provider 的实现原理：

```typescript
// packages/ai/src/providers/faux.ts
export function registerFauxProvider(): FauxProviderRegistration {
    const state = { callCount: 0, ... };
    let responses: AssistantMessage[] = [];

    const provider: ApiProvider<"faux", StreamOptions> = {
        api: "faux",
        stream(model, context, options) {
            const stream = new AssistantMessageEventStream();
            const response = responses[state.callCount];
            // ... 模拟流式事件
            return stream;
        },
        streamSimple(model, context, options) {
            return provider.stream(model, context, options);
        },
    };

    registerApiProvider(provider);
    return {
        setResponses: (r) => { responses = r; },
        getModel: () => ({ ...FAUX_MODEL, api: "faux" }),
        state,
        unregister: () => unregisterApiProviders("faux"),
    };
}
```

这种测试方式的优势：
- **速度快**：无需网络请求
- **可靠**：不会因 API 限流或网络问题失败
- **可预测**：精确控制 LLM 的回复内容
- **覆盖全面**：可以模拟正常回复、工具调用、思考内容等各种场景

### 8.3.3 Coding Agent 测试架构

coding-agent 有专门的测试工具 `test/suite/harness.ts`，用于模拟完整的 Agent 运行环境：

```typescript
// test/suite/harness.ts — 简化的用法
import { createTestHarness } from "./suite/harness.ts";

const harness = await createTestHarness({
    // 使用 Faux Provider
    responses: [
        fauxAssistantMessage("I'll read the file."),
        fauxAssistantMessage([
            fauxToolCall("read", { path: "/tmp/test.txt" }),
        ]),
    ],
});

// 发送提示
await harness.session.enqueueUserMessage("Read the file");
await harness.session.runPrompt();

// 断言
expect(harness.getEvents()).toContainEvent("tool_execution_start");
expect(harness.getEvents()).toContainEvent("message_end");
```

## 8.4 Lockstep 版本发布流程

### 8.4.1 版本管理

所有包使用**相同的版本号**，每次发布同时更新所有包：

| 版本变更类型 | 命令 | 适用场景 |
|-------------|------|---------|
| Patch | `npm run release:patch` | Bug 修复、新功能 |
| Minor | `npm run release:minor` | API 破坏性变更 |

### 8.4.2 发布流程

完整的发布流程由 `scripts/release.mjs` 自动化：

```
1. 检查未提交的变更
2. 版本号提升（所有包）
3. 更新 CHANGELOG 文件
   └── [Unreleased] → [0.75.3] - 2026-01-15
4. git commit + tag
5. npm publish（所有包）
6. 添加新的 [Unreleased] 章节到 CHANGELOG
7. git commit
8. git push --follow-tags
```

### 8.4.3 CHANGELOG 规范

每个包有自己的 `CHANGELOG.md`，遵循 Keep a Changelog 格式：

```markdown
# Changelog

## [Unreleased]

### Added
- New feature description (#123)

### Fixed
- Bug fix description (#456)

### Changed
- Breaking or notable changes (#789)

## [0.75.3] - 2026-01-15

### Fixed
- ...
```

规范要求：
- 每个版本发布后不可修改
- 新条目只添加到 `[Unreleased]` 部分
- 使用预先定义的 section：`Breaking Changes`、`Added`、`Changed`、`Fixed`、`Removed`

## 8.5 贡献指南

### 8.5.1 Issue / PR 门控机制

pi 的 GitHub 仓库使用了自动门控机制来管理贡献：

- **新贡献者**的 Issue 和 PR 会被自动关闭
- 维护者**每天审核**被关闭的 Issue
- 通过 `lgtmi` 评论批准未来的 Issue 提交
- 通过 `lgtm` 评论批准 Issue 提交 + PR 权限

代码文件位于 `.github/workflows/` 下：
- `issue-gate.yml` — Issue 自动关闭
- `pr-gate.yml` — PR 自动关闭
- `approve-contributor.yml` — 维护者批准

### 8.5.2 AGENTS.md — 给 AI 代理的规则

`AGENTS.md` 是一个独特的设计——它是给 AI 编码代理（如当前正在阅读的我们）阅读的说明文件，包含了：

- 代码规范（不使用 `any`、erasable syntax 等）
- 命令规则（`npm run check`、测试等）
- Git 提交规则（仅提交自己修改的文件）
- 贡献门控说明
- 发布流程

## 8.6 章节小结

pi 的工程实践体现了高质量 TypeScript 项目的标准做法：

1. **工具链**：Biome（Lint+格式化）+ tsgo（编译）+ Vitest（测试）
2. **特殊约束**：erasable syntax only、`.ts` 扩展名导入
3. **测试策略**：Faux Provider 实现无网络依赖的可靠测试
4. **版本发布**：Lockstep 版本管理 + 自动化脚本
5. **贡献管理**：门控机制 + AGENTS.md 给 AI 代理看
6. **CHANGELOG**：每个包独立、结构化、不可变的历史记录
